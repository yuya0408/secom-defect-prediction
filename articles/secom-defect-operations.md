これは SECOM 不良予測プロジェクトの3本目、「運用編」である。

- **考察編**：このデータが何を物語っているかを読み解いた話
- **実装編**：読み解いた結果を API・コンテナとして動かせる形にした話
- **運用編（本記事）**：動かし続けるために何を監視し、異常を見つけたらどう動くかの話

実装編で「精度に限界のあるモデルでも、動かせる形にして初めて現場で使う土俵に乗る」と書いた。では土俵に乗せたあと、そのモデルをどう見守るのか。本記事はその続きにあたる。

## 1. なぜ「運用編」が必要だったか

考察編で、SECOM の最大の発見は「品種ミックス」だった。欠損パターンでサンプルを分類すると、不良率がまるで違うグループに分かれる。そして第7章で、見逃せない事実にぶつかった。

**時系列で評価すると、テスト期に「学習時には存在しなかった品種グループ」が次々と現れる。** あるfoldではテストデータの35%が新グループだった。

これは分析上は「品種情報をモデルに組み込めなかった理由」のひとつだったが、運用の視点で読み直すと、もっと重い意味を持つ。**学習したときの世界と、いま流れてくる製品の世界は、時間とともにズレていく**——いわゆるデータドリフトだ。実装編で作った推論APIをそのまま現場に置き続ければ、いつかモデルは「もう知らない世界」を相手に予測することになる。

精度0.6のモデルである。ただでさえ余裕がないのに、土台の前提が崩れたら予測は意味を失う。だから運用編のテーマはこうなる——**モデルが「もう知らない世界」を相手にしていないかを、どう見張るか**。

---

## 2. 何を監視するのか：3層に分けて考える

ドリフトと一口に言っても、見るべきものは1つではない。本プロジェクトでは3層に分けた。

| 層 | 見るもの | ラベルの要否 |
|----|----------|--------------|
| 入力ドリフト | 入ってくるデータの分布が学習時とズレていないか | 不要（即時に分かる） |
| 予測ドリフト | モデルが出す不良確率の分布が変わっていないか | 不要（即時に分かる） |
| 性能ドリフト | 実際の検査結果に対して、予測が当たっているか | **必要（検査結果が出てから）** |

製造現場で重要なのは、**検査結果（真のラベル）は遅れて確定する**という点だ。製品を作った瞬間に「不良だったか」は分からない。検査を経て初めて分かる。だから性能ドリフト（当たっているか）は事後にしか測れない。

ここに監視の構えが決まる。**入力・予測ドリフトで「即時に異常の気配」を察知し、検査結果が溜まったら性能で裏を取る**、という二段構えだ。

---

## 3. 品種ミックスを、そのまま監視指標に転用する

入力ドリフトをどう測るか。ここで考察編の発見がそのまま効いた。

考察編では、欠損率10〜70%の列の欠損パターンで「品種グループ」を定義した。運用でも同じ署名を使う。学習（参照）期に現れたグループの集合を覚えておき、新しく入ってきたデータに対して、**学習時に存在しなかったグループの割合（未知グループ率）**を計算する。

```python
# src/monitoring.py より（要点）
def missing_pattern_signature(df, flag_cols):
    """欠損パターンからサンプルごとのグループ署名（"010..." 形式）を作る。"""
    miss = df[flag_cols].isna().astype(int)
    return miss.apply(lambda row: "".join(map(str, row.values)), axis=1)

def unknown_group_rate(reference_signatures, new_signatures):
    """学習時に存在しなかったグループ署名を持つ新データの割合。"""
    known = set(reference_signatures)
    return sum(1 for s in new_signatures if s not in known) / len(new_signatures)
```

予測分布のズレには **PSI（Population Stability Index）** を使った。学習時のスコア分布を分位点でビンに切り、新データの各ビン比率と比べる定番の指標で、慣用的な目安は「0.1未満なら安定、0.1〜0.25は要注意、0.25超は大きな変化」。

外部のドリフト監視ツール（Evidently など）を使う選択肢もあったが、今回は依存を増やさず、素の numpy/pandas で書いた。中身がブラックボックスにならず、何を計算しているかを自分で説明できる状態にしておきたかったからだ。

## 4. 「動かして見せる」：時系列リプレイ

監視ロジックを書いても、流れてくるデータがなければ意味がない。だが SECOM は2008年で完結した凍結データで、本番ラインがあるわけではない。

そこで、**凍結データの上でドリフトを再現する**ことにした。検査タイムスタンプで全データを月ごとに分け、最初の数か月を「参照（学習）期」、以降の月を「本番運用月」として順にモデルへ流す。リプレイである。

```bash
python -m scripts.replay_drift
# => reports/drift_report.md と図を生成
```

本プロジェクトのデータは2008年7月〜10月の4か月。**7〜8月を参照期としてモデルを学習し、9月・10月を本番月として流した**。

凍結データには、本番ラインにはない強みがひとつある。**全期間の検査結果（ラベル）が揃っている**ことだ。本番では遅れて確定する性能ドリフトを、ここでは即座に測れる。デモのための割り切りだが、おかげで3層すべてを通しで実演できる。

結果がこれだ。

![時系列リプレイによるドリフト指標（月別）](/images/secom-defect-operations/fig7_drift_replay.png)

| 月 | 未知グループ率 | 予測PSI | PR-AUC | 判定 |
|----|----------------|---------|--------|------|
| 2008-09 | 9.2% | 0.279 | 0.202 | RED |
| 2008-10 | **35.9%** | 0.397 | 0.141 | RED |

未知グループ率が **9.2% → 35.9%** と月を追って増えている。考察編で見た「テスト期に新グループが現れる」現象が、運用指標として目の前で動いた瞬間だった。10月にはPR-AUCも参照期の基準（0.18）を下回り、性能の劣化まで観測できた。

正直に言えば、両月とも「赤」で、緑や黄の穏やかな段階を経ずにいきなり危険水準に入っている。これは取り繕わずに書いておきたい。**9月の時点で、モデルが学習した7〜8月の世界とは、もうかなりズレていた**ということだ。考察編で見た「9月に不良率が3%を切る」激しい変動とも符合する。

## 5. 失敗の記録：自分の答案を自分で採点していた

ここで、実装の途中でやらかした方法論のミスを残しておきたい。考察編が「リークに気づいて測り直した」記録だったのと、同じ精神でだ。

最初に書いたコードは、参照期の「基準性能」をこう計算していた。

```python
model.fit(X_ref, y_ref)                       # 参照データで学習
ref_scores = model.predict_proba(X_ref)[:, 1] # ← 同じ参照データを採点
baseline_pr_auc = average_precision_score(y_ref, ref_scores)
```

これを動かすと、基準 PR-AUC が **1.0000**、予測PSIが **3.8〜4.0** という、見るからにおかしな値が出た。

原因は、**学習に使ったのと同じデータをそのまま採点していた**こと。Random Forest は学習データをほぼ暗記するので、訓練行への予測は不良≈1.0・合格≈0.0 にきれいに張り付く。その結果、

- 暗記したデータ上では完璧に分類できるので、基準 PR-AUC が 1.0 になる。当然その後の全月が「基準割れ」になり、性能の判定が無意味化する。
- 暗記スコアは0と1に張り付いた極端な分布になる。一方で本番月のスコアは0.1前後に穏やかに広がる。この2つを比べれば「激しく違う」のは当たり前で、PSIが爆発する。だがその違いは過学習が生んだ見せかけで、実ドリフトではない。

修正はシンプルで、**交差検証の out-of-fold 予測**に置き換えた。データを5分割し、各サンプルを「それを学習に使っていないモデル」で採点する。

```python
ref_scores = cross_val_predict(
    make_model(), X_ref, y_ref,
    cv=StratifiedKFold(5, shuffle=True, random_state=42),
    method="predict_proba",
)[:, 1]
baseline_pr_auc = average_precision_score(y_ref, ref_scores)
```

これで基準 PR-AUC は **0.1825** に落ち着いた。考察編の厳密評価値 0.1346 と同じく「ランダム水準の約2倍」という低い値で、桁が合う。基準が現実的になり、PSI も意味のある値に戻った。

「自分の答案を自分で採点すれば満点が出る」——言われれば当たり前なのに、コードにすると気づきにくい。考察編のリークと根は同じで、**評価の独立性を一箇所でも崩すと数字は簡単に嘘をつく**、という教訓をもう一度なぞった格好だった。

## 6. 検知したら、どう動くか

監視の本番は「異常を見つけたあと」だ。本プロジェクトでは、深刻度を緑/黄/赤の3段階で判定し、段階ごとに動きを変える設計にした。判定はコードで行うが、**実際のアクションは人間が承認する**ことを前提にしている。

```python
# src/monitoring.py より
RECOMMENDED_ACTIONS = {
    "green":  "通常運用。指標を記録するのみ。",
    "yellow": "推論は継続しつつ通知・ダッシュボード警告。レスポンスに drift_warning を付与。",
    "red":    "推論は返すが信頼度を下げ、該当ロットを全数検査へ。再学習を起票（自動実行しない）。",
}
```

設計で意識したのは3点だ。

**赤でもAPIを止めない。** 製造ラインの検査補助は、止まると現場が困る。だから赤でも予測は返す。ただし「これは信用しすぎるな」という信号（`drift_warning`）を一緒に出し、該当ロットは自動振り分けに使わず全数検査へ回す。フェイルセーフは「止める」ではなく「人の目を増やす」側に倒した。

**自動で再学習・自動でデプロイしない。** 赤になったら再学習を「起票」するが、自動では走らせない。新しいモデルは必ず時系列ホールドアウトで評価し、現行と比べて良いか同等のときだけ昇格させる。悪化していたらロールバックする。

**再学習は目的ではなく手段。** ここが製造現場上がりとして一番こだわった点だ。考察編で「品種情報を足しても精度は上がらなかった」ことを思い出してほしい。このデータでは、再学習しても精度が上がるとは限らない。だから監視の主目的を「精度を上げること」に置くのは筋が悪い。置くべきは、**入力が学習時から乖離してモデルが無意味化していないかの検知と、しきい値の再調整・再学習を起票するかの判断**だ。

## 7. これは「実演」であって「本番監視」ではない

最後に、誇張を避けるために線を引いておきたい。

本記事でやったのは、**凍結データの上でドリフト監視機構を動かして見せた**ことだ。本番の製造ライン監視とは、決定的に違う点がいくつもある。

- 本番では検査ラベルが遅れて確定する。ここでは全期間揃っているので性能ドリフトを即座に測れたが、これは凍結データだからこその割り切りだ。
- 本番ではアラート・ダッシュボード・再学習パイプラインといった常時稼働の基盤が要る。本記事は判定ロジックと実演までで、その基盤までは作っていない。「設計はこう」という思想をコードと文章で示すに留めた。
- 仮にこのモデルを Hugging Face Spaces などに載せてデモ公開しても、そこに流れるのはデモ利用者のトラフィックであって、製造ラインの実分布ではない。

それでも、**監視機構が確かに動き、考察編で見つけたドリフトを指標として捉えられる**ことは示せた。精度0.6のモデルを現場に置き続けるなら、「精度を上げる」より先に「壊れていないかを見張る」仕組みが要る——その入口は作れたと思う。

## 結び

3本を通して言いたかったのは、ひとつのことに尽きる。**ノートブックでスコアを出すことと、モデルを現場に置き続けることの間には、いくつもの段差がある**、ということだ。

考察編では「データを読む」段差を、実装編では「動かせる形にする」段差を、そして運用編では「動かし続ける」段差を、それぞれ手で確かめた。どれも派手な技術ではない。リークを疑い、import パスを揃え、自分の答案を自分で採点していないかを確かめる——地味な確認の積み重ねだった。

精度0.6のモデルは、それ単体では頼りない。だが、何を見張ればいいかを決め、異常を見つけたときの動き方を用意しておけば、現場に「置き続ける」ことはできる。製造業にいた頃に学んだ「変化点を見張る」という感覚は、モデルの運用にもそのまま効いた。データを読む力、動かす技術、そして見守る仕組み。その3つが揃って、ようやく分析は現場に根づくのだと思う。
